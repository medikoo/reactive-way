The way we build web applications of today in many cases do not touch the potential that modern engines have to offer

Popular frameworks are still prisoners of support of ES3 based engines. We waste a lot of time on micro-optimizations and we like to repeat the patterns with which we developed web applications 5, 10 even 15 years ago, as this is what we're familiar with and safe with

Sometimes it seems that we didn't notice now drastically hardware that runs our apps improved, and that todays browsers are very different beasts then one we've dealt with years ago.

Typical website delivery architecture

Before I'll get into raw meat, I'd like us to have a quick and very general overview of how we deliver applications.

We can distinguish three important components:

A persistent storage for all the data, a server which runs business logic, responds do requests, delivers the interface, and of course a client right in front of a consumer of application

Over a decade ago..

Client browser

  • HTML renderer with basic style and scripting support
  • All user requests directed to server

Server process

  • HTTP server
  • Scripts, reevaluated on each request
  • Data, read and stored to physical storage on each request

What was the role of those components not that long ago (?)

Delivery of applications involved constant and repeated data transformations

Typical request handling looked that, when request came from a client, CGI or PHP script was reevaluted, this script unserialized data from physical storage, produced expected response and it was gracefully killed.

Exactly same loop was repated on another request and so on, so on

Serialize & Unserialize

Today

Client browser

  • Host of complete applications

Server

  • Host of ongoing application processes

Today with Node.js, we not only have light process that's server of an application but it also can be a process that handles business logic of our application, and it's not we that reinitialize that process on each request.

Still quite often we try to deliver applications repeating same patterns we learned to work with over a decade ago

e.g. in SPA applications we generate HTML strings, inject it via innerHTML, then search for created elements with some querySelector mechanism to provide additional behaviors.. and all of that happenswithin same, one memory process

Same with databases, if we think of replicating some logic that was handled by database storage on server, we assume we need to bring some SQL or other external language to be able to process our data efficiently

What if..

What if we'll try to unify our architecture in that one process

We can create DOM directly, and omit serialize/unserialize HTML step

But it's not only about how we generate views. We can also take most of relational data handling logic step (one that usually was part of database engine, like SQL) to be a part of our process, and leave persistent layer just with primitive key/value forms

Technically we can make our application process to be a living organism, that works with the rules we defined for it

Reactive programming

When you start to think and work that way, you instinctively get into reactive programming

What is reactive programming?

To clarify I'm not going to talk about Facebook's React, or any other popular stack with "reactive" in its name that recently emerged

Let's talk about Reactive in simplest, purest form with no framework noise

Compose flow
like in a spreadsheet application

Reactive programming is about assuring bindings between our mutable data, transparantly behind the scenes

Plain imperative programming

var a, b, c;
a = b + c;
b = 2;
c = 3;
console.log(a); // ?NaN

is obviously not reactive

How to achieve reactiveness in JS

1. Recalculate after change

var a, b, c;
var refresh = function () { a = b + c; };
b = 2;
c = 3;
refresh();
console.log(a); // 5

b = 4;
refresh();
console.log(a); // 7

This how reactivity works in most popular reactive tools today, and it's actually how we can force reactiveness to not really reactive architectures

This is how we update state in many SPA applications today, e.g. on change we regenerate new HTML, replace with new one, and Make sure old one ready to be garbage collected

Still I would call it an emulation of reactive setup, we can do better...

How to achieve reactiveness in JS

2. Introduce observable atomic values

var Observable = require('observable-value');

var b = new Observable()
var c = new Observable();
var a = b.add(c); // b + c

b.value = 2;
c.value = 3;
console.log(a.value); // 5

b.value = 4;
console.log(a.value); // 7

We can introduce observable atomic values with dedicated interface

This is as fine grain reactive as we can be, and this observable interface exposes many more methods and functions which allow to construct desired reactive chains.

Stil, you may be sceptic, seeing this code, wondering whether it means that such observables should be created manually, it seems as a bit verbose approach

Nothing like that, it's more about using utilities that provide such interfaces out of a box and understand them ouf of a box.

DOM is a perfect match

var exampleForm = form(table(tbody(

exampleForm.oninput = function () {

There's a domjs engine which allows to compose DOM view in readable way

Objects like a, b or c are usually coming from other utlities, they're not created as observables on a spot

Reactive notation is self-explanatory

Basic example of real world view

h1(company._name);

_if(not(company._isSubmissionComplete),
  p("Please complete application"));

h2(company._name, " Branches:");

// Reactive list:
ul(company.branches, function (branch) {
  return li(
    { class: _if(branch._isMain, 'main') },
    h3(branch._name), p(branch._address)
  );
});

In that example data objects already come with observable interface implied

company.name returns plain string value, but company._name returns an observable of company's name property.

_ (underscore) choice may feel controversial. Still I prefered that over $. There's also explicit way via getObservable('name'), but using it, would introduce a lot of noise to the templates

There are functions which help compose reactive configurations in straightforward way, e.g. this paragraph, will be included in DOM, only if company.isSubmissionComplete is falsy

Plain text can be mixed with reactive one

There is also reactive list (in that case set) interface, which produces auto updatable behavior

Values for attributes can also be configured reactive way

domjs provides an utility that compiles HTML into domjs notation

Utilities for reactive configurations

  • Value: github.com/medikoo/observable-value

    (inc. comparision, arithmetic, logical and conditional utilities)

  • Set: github.com/medikoo/observable-set

    (inc. unions, intersections, complements, filters, maps and array representations)

  • Map: github.com/medikoo/observable-map

    (inc. filters, maps, subsets and set representations)

  • Array: github.com/medikoo/observable-array

    (inc. filters, maps, slices and sorts)

Important to note: They're all use case agnostic

On their own they do not provide or depend on any DOM bindings, they can be used to configure viewless reactive configurations in server-side processes

With this utitlities you can easily compose any reactive setup for whatever purpose

In-Memory Database

Views is where reactivity really shines, but the most powerful part of system we're working with is In-Memory Database written in plain JS

Persistently our data is stored in representatin of primitive key/value pairs, but it doesn't mean we process our applications data working with that notation

On top of that we have an engine, which provides a means to manipulate, filter, search, index the data in probably most natural (for JavaScript language) way you can imagine

dbjs: github.com/medikoo/dbjs

var Database = require('dbjs');
db = new Database();

Primitive types

Boolean, Number, String, DateTime, RegExp, Function

Work similar to JS constructors

db.Number('343'); // 343

.. but they're more strict

db.Number('foo'); // TypeError

Initially (in 2011) it was made for a purpose of offline HTML5 application, a registration system, that presented data in different configurations, tables, filters and provide a mean to search over collected data

Over time it grown up to be much more than that, and a lot of work was made to improve its perfomance and desgin. At least twice it was rewritten from scratch.

What is dbjs really? If I would have to put it one sentence: It's a database of events which on a front composes into very familiar object structures. dbjs it's also designed to be as seamless with the langauge as possible

Creating a database instance is like creating a new realm for your data.

Primitive Type extensions

db.String.extend('ShortString',
  { max: { value: 3 } });

db.ShortString('foo'); // foo
db.ShortString('foobar'); // TypeError

db.String.extend('ZipCode',
  { pattern: { value: /^\d{2}-\d{3}$/ } })

db.ZipCode('00-801'); // '00-801'
db.ZipCode('whatever'); // TypeError

Any Type can be extended to more custom one

Example on right is actually an equivalent of ShortString extends String ES6 classes syntax, and here extend function shares similar signature as Object.create as it accepts descriptors for properties. It's however about dbjs (no ES5) level descriptors that describe meta characteristics of given property

db.Object an object type

var obj = db.Object({ foo: 'bar', bar: 34 });

obj.foo; // 'bar'
Object.keys(obj); // ['foo', 'bar']

db.Object is an extension of Map type

obj.set(34, 'Value for 34');
obj.set(true, 'Value for boolean true');
obj.set('str', 'Value for str');

obj.get(true); // 'Value for boolean true'

All string keys are exposed as enumerable properties

Object.keys(obj); // ['foo', 'bar', 'str'];

There's of course object type

It may seem to work exactly as native JS constructor, but there's of course significant difference. With very simple (behind a scene) setup, all those operations can easily be saved and restored, but we'll get to that later

Objects are a combinations of object dictionaries and instances of Map (type introduced with ECMAScript 6)

Turning it around we can say it's a map with all string keys exposed as object properties

Object extensions

db.Object.extend('Person', {
  firstName: { type: db.String, required: true },
  lastName: { type: db.String, required: true },
  fullName: { type: db.String,
    value: function () {
      return this.firstName + " " + this.lastName;
    }
  }
});

var john = new db.Person({ firstName: "John",
  lastName: "Smith" });

john.fullName; // "John Smith"

Via Object type extensions we can define classes with which we model our application data

We define our data with descriptors of meta characteristics

We can also define dynamic properties. They do not need any other handling to work as expected. dbjs internally uses esniff low footprint ECMAScript parser, and knows on which properties this property depends. In case of more complex dependency setup, that can't be reliably statically parsed, you can make use of _observe registration function, provided as first arguments, and call it wih any observable which value affects the result.

Dynamic property can depend on other dynamic properties, it's one of the most powerful aspects of this engine.

Instances can be created in a way you can guess. If you wonder, new keyword is optional

Object extensions #2

More sophisticated example

db.Person.extend('Patient',
  { birthDate: { type: db.DateTime } });

db.Person.extend('Doctor', { patients:
  { type: db.Patient, multiple: true,
   reverse: 'doctor', unique: true } });

var drHouse = new db.Doctor({ firstName: "Gregory",
  lastName: "House" });

var henry = new db.Patient({ firstName: "Henry",
  lastName: "Smith",
  birthDate: new Date(1977, 0, 3) });

drHouse.patients.add(henry);
john.doctor; // drHouse

We can of course define multiple property values. They'll be represented with sets. It's worth to note here, that dbjs handles two types of collection values map and set, and mentioned before atomic primitive values, which can represent any key or value of a map, or be part of a set

We can assure uniqueness of property values. Here it means that whenever we try to assign a patient to a second doctor exception will be thrown.

In case of object to object relations, we can define to have property holder exposed at it's value object. In this example doctor, to which patient was assigned will be also revealed at doctor property on patient.

Still, it's not everything, in this presentation I introduce features very selectively, there are many things I'm not able to cover in this short run

Meta configurations are reactive

var birthDateDesc = db.Patient.prototype
  .getOwnDescriptor('birthDate');

john.birthDate;             // [Date 1977, 0, 3]

birthDateDesc.type = db.Number;
john.birthDate;             // 221094000000

birthDateDesc.type = db.DateTime;
john.birthDate;             // [Date 1977, 0, 3]

At any time we can take the descriptor of any property, change chosen characteristics and it will have instant impact on the data

Still it's not destructive for the data. Assigning back previous value will bring back previous state

So, how all those changes are represented for persistent layer (?)

In a table there's a representation of al updates that I did in previous examples. It's not shown in this table, but each update is marked with a unique timestamp of microseconds accuracy. Directly we of course never work with this notation

Key column represents ids of entities that were updated.

Newly crated object are marked with universally unique identifiers. It's based on current microseconds timestamp and random string

Meta (descriptor) properties are identified with $

First character of serialized value tells us what is it.

On reinitialization those events can be unserialized in any order to restore the state

Real application demo
elomas.gob.ar

Application helps citizens of Lomas de Zamora (province of Buenos Aires) to register the business

Whole application model is defined with help of dbjs engine. It's complex, model definition covers over 12 000 properties.

Each view is configured reactive way with help of domjs. Apart of exceptions of few static pages of public website, it's all pure direct DOM generation, with no innerHTML involved

There's a simulation page where you can see demo of a initial form visible to registered user. Based on area in which business activity would be pursued and other business characteristics it is decided what would be the cost of registration, what conditions have to be met and which documents are required by applicant for successfully registration

Most of meta data involved in this decision making, can be edited in special administration panel, described e.g. here Example 1 Example 2, Example 3. Changes made in those panels do not require restart of a server, they immediatelly affect the state of the data.
